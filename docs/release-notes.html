Release Notes for OpenLaszlo 5.0

OpenLaszlo 5.0 is a ground-up modernization of the OpenLaszlo platform. It removes the Java dependency entirely, introduces in-browser compilation, and stays byte-for-byte compatible with the original OpenLaszlo 4.9 compiler. Your existing LZX applications compile and run unchanged.

No more Java

The LZX-to-DHTML compiler has been re-implemented in TypeScript and runs on Node.js. The entire toolchain and the DHTML runtime are now Java-free: there is no JDK, no servlet container, and no JSP anywhere. Building and serving an OpenLaszlo application no longer requires a Java installation.

In-browser compilation

The same compiler is also packaged as a small, self-contained browser bundle. Using a Service Worker, OpenLaszlo 5.0 can intercept a request for an .lzx application, compile it to DHTML directly in the browser, run it, and cache the result — with no build step and no server round-trip.

100% compatibility with OpenLaszlo 4.9

The new compiler is differential-tested against the original Java 4.9 compiler and produces byte-for-byte identical output — across the documentation example corpus, the Laszlo Explorer showcase, and complete applications, in both production and debug builds. Existing LZX source code compiles without modification, so upgrading from OpenLaszlo 4.9 requires no changes to your applications.

Release Note: the only compiler capability not yet reproduced is the optional backtrace debugging-instrumentation mode — a development-time aid that is never included in a deployed application. All standard production and debug output is identical to OpenLaszlo 4.9.

Hosting options

Static hosting (no application server)

Because applications can be compiled in the browser, OpenLaszlo 5.0 can be hosted on any plain static web server or content-delivery network. A small starter page registers a Service Worker that compiles and runs your .lzx applications on demand and caches the compiled output. No dynamic backend is required to serve and run an application.

Server hosting (Node.js)

A lightweight, dependency-minimal Node.js server is also included. It compiles applications on the server with on-disk caching, serves the runtime and assets, and supports dynamic, server-driven backends — for example, the persistent-connection chat demo running over WebSockets. Choose this option when you want server-side compilation or live server data.

Bonus: the Dashboard demo, ported from OpenLaszlo 3.x

As a bonus, the classic Dashboard demo has been ported forward from OpenLaszlo 3.x and now runs in DHTML under OpenLaszlo 5.0.

Migrating your Applications to OpenLaszlo 4.9.1

For those of you who have already upgraded your applications to OpenLaszlo 4.2 or higher, no further work is needed. You should just start using OpenLaszlo 4.9.1.

To migrate your 4.0.X or 4.1.1 applications, we strongly suggest that you refer to this wiki page: Runtime_Differences. This page discusses the changes required by SWF9 and also provides a methodology for upgrading your application. It is very important that you run the automated conversion scripts in the recommended order, should you choose to take advantage of them. To review the changes in incremental releases, please refer to the OpenLaszlo Archive

Workaround for Input Text Unexpected Behavior with onfocus Handler

In some cases, the onfocus event fires too early, causing unexpected input text characters to be selected as it does in this example: 

  <canvas layout="axis: y">
    <inputtext text="abcd" width="100" bgcolor="0xcccccc"/>
    <inputtext text="abcd" width="100" bgcolor="0xffcccc" onfocus="this.setSelection(0, 1)"/>
   </canvas>

To work around this issue, you can create a cover view that handles initial clicks, as shown here for the above example: 

  <canvas layout="axis: y">
    <inputtext text="abcd" width="100" bgcolor="0xcccccc"/>
    <view bgcolor="0xffcccc"> <inputtext name="field" text="abcd" width="100">
      <handler name="onfocus"<
        parent.cover.setVisible(false);
        this.setSelection(0, this.getText().length);
      </handler>
      <handler name="onblur">
        parent.cover.setVisible(true);
      </handler>
      </inputtext>
      <view name="cover" width="100%" height="100%" onclick="LzFocus.setFocus(parent.field)" />
    </view>
  </canvas>

Deprecation Notice: Attribute Values on Data Element Will Be Coerced to String in a Future OpenLaszlo Release

Currently, if you set an attribute value for a data element, the value is stored as it is set. This will no longer be true in future OpenLaszlo releases, where the value will be coerced to a String value. If you need to set non-String values for data elements, you should use the setUserData method as shown in this example where node is an LzDataElement:
node1.setUserData('key1', val1); assertEquals(val1, node1.getUserData('key1'), 'getUserData key1')
